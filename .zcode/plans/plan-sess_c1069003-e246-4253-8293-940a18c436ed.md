## Root Cause

`TypeError: Cannot read properties of undefined (reading 'exports')` is a **CommonJS/ESM interop failure**. Your `package.json` has `"type": "module"`, so all output (including `out/main/index.js`) is ESM. `better-sqlite3` and `xlsx` work because electron-vite auto-inserts a `createRequire` shim for them:

```js
const require2 = __cjs_mod__.createRequire(import.meta.url);  // line 12
const Database = require2("better-sqlite3");                    // line 13 â€” WORKS
```

But `@supabase/supabase-js` and `electron-updater` are also CommonJS (`main: dist/index.cjs` / no `type:module`), and electron-vite does **NOT** give them the same shim. When `sync.ts` runs `import { createClient } from '@supabase/supabase-js'`, Node's default ESMâ†’CJS interop chokes on supabase's internal `module.exports` pattern â†’ the error.

The previous developer correctly identified the symptom but applied the wrong fix (commenting everything out instead of applying the same `createRequire` shim that better-sqlite3 already uses).

## The Fix (2 layers)

### Layer 1: Force CJS deps through `createRequire` (the real fix)

**`electron.vite.config.ts`** â€” Remove the brittle manual `external` array and add a Rollup `noExternal` shim won't work for externalized deps. Instead, use electron-vite's built-in approach: configure `build.rollupOptions.external` to a function that keeps them external, AND add a tiny virtual shim.

The cleanest approach: **load supabase/updater via `createRequire` directly in source code**, matching the pattern electron-vite already uses. In `src/main/sync.ts`:

```ts
import { createRequire } from 'node:module'
const require = createRequire(import.meta.url)
const { createClient } = require('@supabase/supabase-js')  // CJS load â€” no interop error
```

Same pattern for the auto-updater (currently undefined `getAutoUpdater`).

**`electron.vite.config.ts`** â€” simplify the manual `external` array (it's currently hand-listing deps that `externalizeDepsPlugin()` already handles, which is redundant). Keep it clean.

### Layer 2: Uncomment & wire everything back up

1. **`src/main/index.ts`** â€” uncomment sync engine + auto-updater init
2. **`src/main/ipc.ts`** â€” uncomment `registerSyncIpc()`
3. **`src/preload/index.ts`** â€” uncomment sync + auto-update API methods
4. **`src/renderer/src/App.tsx`** â€” uncomment sync status indicator + polling
5. **`src/renderer/src/components/SettingsScreen.tsx`** â€” uncomment Supabase config UI

### Layer 3: Fix latent bugs found during investigation

- **`db.ts:749`** â€” `splitRow` query has 2 placeholders (`created_at >= ? AND shop_id = ?`) but only passes 1 arg. Missing `shopId`. Will crash dashboard.
- **`src/main/index.ts`** â€” `getAutoUpdater` is referenced but never defined/imported. Need to add a proper `electron-updater` loader using the same `createRequire` pattern, plus wire up `autoUpdater` from `electron-updater`.

### Layer 4: Make sync gracefully no-op (since Supabase tables don't exist yet)

Since the user has a Supabase project but **no tables**, sync must not crash when unconfigured. It already does this (`initSupabase` returns early if no URL/key in settings). The sync runs only after the user enters their Supabase URL + anon key in Settings. The SQL to create the cloud tables can be provided as a reference but isn't wired into the app.
